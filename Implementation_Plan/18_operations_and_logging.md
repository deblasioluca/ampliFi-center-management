# 18 — Operations: Logging, Scripts, Linux Service

This module specifies the operational scaffolding the implementer must ship: a
robust logging stack, lifecycle scripts (start / stop / restart / status / install),
systemd unit files, and the install-as-a-service workflow on a Linux host.

The reference repo `sap-ai-consultant` already establishes a logging pattern; reuse
its conventions when the PAT is provided. Until then, implement the spec below.

## 18.1 Logging stack

### 18.1.1 Library + format

- **Backend (Python)**: `structlog` on top of stdlib `logging`. JSON formatter in
  prod, key-value renderer in dev. Single configuration in
  `backend/app/infra/logging.py`.
- **Frontend (Astro server)**: `pino` with the same JSON shape.
- **Workers (Celery)**: same structlog config; Celery hooks (`task_prerun`,
  `task_postrun`, `task_failure`) attach `task_id`, `task_name`, `run_id`.

Common log envelope:

```json
{
  "ts": "2026-04-25T12:34:56.789Z",
  "level": "info",
  "service": "backend",
  "env": "prod",
  "request_id": "...",
  "user_id": 42,
  "trace_id": "...",
  "span_id": "...",
  "event": "wave.lock",
  "wave_id": 17,
  "duration_ms": 134
}
```

Required correlation fields:
- `request_id` (per HTTP request, generated by middleware).
- `trace_id` / `span_id` from OpenTelemetry context.
- `task_id` for Celery work.
- `actor` (user_id or token id) for audited events.

### 18.1.2 Log levels (and what goes where)

| Level | Use |
|---|---|
| DEBUG | Verbose dev info; never in prod files unless explicitly enabled |
| INFO  | Normal operational events: requests handled, jobs started/finished, OData refresh batch counts |
| WARN  | Recoverable issues: retried API call, validation rejected rows, cache miss above threshold |
| ERROR | Caught exceptions with context, failed jobs, LLM provider errors |
| CRITICAL | Anything requiring on-call: DB unreachable, Celery broker down, secret missing, daily cost cap breached |

### 18.1.3 Files & rotation

Default file layout (configurable via env):

```
/var/log/amplifi-cleanup/
├── backend.log          # FastAPI + structlog JSON, rotated daily, 14 days retained
├── backend-error.log    # ERROR+ only; 30 days
├── frontend.log         # Astro/Node
├── workers.log          # Celery workers (one combined file or one per queue)
├── nginx-access.log     # if reverse proxy installed
├── nginx-error.log
└── audit/
    └── audit-YYYYMM.log # mirror of cleanup.audit_log writes for offline forensics
```

Rotation: `logrotate` config installed alongside the service (see §18.4).

```
/etc/logrotate.d/amplifi-cleanup:
/var/log/amplifi-cleanup/*.log {
    daily
    rotate 14
    compress
    missingok
    notifempty
    copytruncate
    create 0640 amplifi amplifi
}
/var/log/amplifi-cleanup/backend-error.log {
    daily
    rotate 30
    compress
    missingok
    notifempty
    copytruncate
    create 0640 amplifi amplifi
}
/var/log/amplifi-cleanup/audit/*.log {
    monthly
    rotate 84       # 7 years
    compress
    missingok
    notifempty
    copytruncate
    create 0640 amplifi amplifi
}
```

### 18.1.4 What MUST be logged (per event)

- **HTTP requests**: method, path, status, duration_ms, request_id, user_id (if any),
  request size, response size. Headers only at DEBUG (and never auth headers).
- **DB queries** (slow only — > 200ms): rendered SQL with bind values redacted, duration.
- **OData calls**: entity_set, $top/$skip, status, duration, rows fetched, retries.
- **LLM calls**: provider, model, prompt_hash, tokens_in, tokens_out, latency,
  cost_usd, cache_hit. **Never** raw prompt or response by default; admin opt-in
  flag enables verbose for one request.
- **Email sends**: template, recipient_hash, status, smtp_response_code.
- **Celery tasks**: task_name, task_id, run_id, args_summary, state, duration,
  retries.
- **Audit events** (mirror `cleanup.audit_log` writes): action, entity, actor,
  before/after diff hash.

### 18.1.5 Sensitive data redaction

A central redactor is applied by the structlog processor:

- Pydantic `SecretStr` fields → `***`.
- Known keys: `password`, `api_key`, `client_secret`, `authorization`, `cookie`,
  `set-cookie`, `x-api-key`, `csrf_token` → `***`.
- Email addresses are kept; long free-text user input is truncated to 256 chars in
  logs (full content lives in DB if needed).

CI lint rule grep-checks logging code for raw `password`, `Authorization`, etc.

### 18.1.6 Centralised aggregation (optional but recommended)

The implementer SHOULD support shipping logs to a centralised system:

- Filebeat / Vector → Elasticsearch / OpenSearch / Loki.
- Or stdout-only mode for container orchestrators (Kubernetes / docker logging
  driver).

Configuration is env-driven (`LOG_TARGET=file|stdout|both`). Default in the systemd
service is `both` so journald + files are populated.

### 18.1.7 Endpoint: live log tail (admin)

`/admin/logs/tail?file=backend&level=ERROR&since=10m` — streams (SSE) the requested
slice. Backed by reading the file with rotation-aware tail logic. Restricted to
admin role; rate-limited.

## 18.2 Lifecycle scripts

Shipped under `scripts/` at the repo root. All scripts:

- Use `set -euo pipefail`.
- Log to stderr with timestamps; use `--quiet` to suppress.
- Honour environment variables for paths (`AMPLIFI_HOME`, `AMPLIFI_USER`,
  `AMPLIFI_VENV`).
- Are idempotent.

### 18.2.1 `scripts/install.sh`

Installs the application as a Linux service.

```bash
#!/usr/bin/env bash
# Usage: sudo ./scripts/install.sh [--user amplifi] [--home /opt/amplifi-cleanup]
# Installs systemd units, creates user, sets permissions, enables services.

set -euo pipefail
USER="${AMPLIFI_USER:-amplifi}"
HOME_DIR="${AMPLIFI_HOME:-/opt/amplifi-cleanup}"

# 1. system user
id -u "$USER" >/dev/null 2>&1 || useradd --system --home "$HOME_DIR" --shell /bin/bash "$USER"

# 2. directory layout
install -d -o "$USER" -g "$USER" "$HOME_DIR"
install -d -o "$USER" -g "$USER" /var/log/amplifi-cleanup/audit
install -d -o "$USER" -g "$USER" /var/lib/amplifi-cleanup/storage

# 3. virtualenv + dependencies
sudo -u "$USER" python3 -m venv "$HOME_DIR/.venv"
sudo -u "$USER" "$HOME_DIR/.venv/bin/pip" install -e backend

# 4. frontend build
( cd frontend && npm ci && npm run build )
install -d -o "$USER" -g "$USER" "$HOME_DIR/frontend"
cp -r frontend/dist "$HOME_DIR/frontend/"

# 5. .env (preserve existing)
[[ -f "$HOME_DIR/.env" ]] || cp .env.example "$HOME_DIR/.env"
chown "$USER:$USER" "$HOME_DIR/.env" && chmod 0600 "$HOME_DIR/.env"

# 6. systemd
install -m 0644 systemd/amplifi-cleanup-backend.service /etc/systemd/system/
install -m 0644 systemd/amplifi-cleanup-frontend.service /etc/systemd/system/
install -m 0644 systemd/amplifi-cleanup-worker.service  /etc/systemd/system/
install -m 0644 systemd/amplifi-cleanup-beat.service    /etc/systemd/system/
install -m 0644 systemd/amplifi-cleanup.target          /etc/systemd/system/
install -m 0644 logrotate/amplifi-cleanup               /etc/logrotate.d/amplifi-cleanup

systemctl daemon-reload
systemctl enable --now amplifi-cleanup.target

echo "Installed. Edit $HOME_DIR/.env and run scripts/restart.sh"
```

### 18.2.3 `scripts/start.sh` / `stop.sh` / `restart.sh` / `status.sh`

Thin wrappers around `systemctl <verb> amplifi-cleanup.target` with friendly output.

```bash
# scripts/start.sh
sudo systemctl start amplifi-cleanup.target
sudo systemctl --no-pager status 'amplifi-cleanup-*' || true
```

`status.sh` additionally prints:
- service states
- last 20 lines of `journalctl -u amplifi-cleanup-backend.service`
- DB / Redis / LLM / SMTP probe (`curl /readyz` and exit 1 if not ready)

### 18.2.4 `scripts/upgrade.sh`

```bash
# Pulls latest, runs migrations, rebuilds frontend, restarts. Health-checks before
# declaring success; on failure rolls back via the previous git tag.
```

### 18.2.5 `scripts/backup.sh` / `scripts/restore.sh`

- `pg_dump` to `${BACKUP_DIR:-/var/backups/amplifi-cleanup}` with date stamp.
- Tar the storage volume (`/var/lib/amplifi-cleanup/storage`).
- Encrypt with GPG when `BACKUP_GPG_RECIPIENT` is set.
- `restore.sh` undoes a chosen archive after a confirmation prompt.

### 18.2.6 `scripts/dev.sh`

Convenience for local dev: `docker compose up -d`, follow logs, tear down on Ctrl-C.

## 18.3 Systemd units

`systemd/amplifi-cleanup.target`:

```ini
[Unit]
Description=ampliFi cost center cleanup (full stack)
Wants=amplifi-cleanup-backend.service amplifi-cleanup-frontend.service amplifi-cleanup-worker.service amplifi-cleanup-beat.service
After=network-online.target postgresql.service redis.service

[Install]
WantedBy=multi-user.target
```

`systemd/amplifi-cleanup-backend.service`:

```ini
[Unit]
Description=ampliFi cleanup backend (FastAPI / Uvicorn)
PartOf=amplifi-cleanup.target
After=network-online.target postgresql.service redis.service

[Service]
Type=simple
User=amplifi
Group=amplifi
WorkingDirectory=/opt/amplifi-cleanup
EnvironmentFile=/opt/amplifi-cleanup/.env
ExecStart=/opt/amplifi-cleanup/.venv/bin/uvicorn app.main:app \
    --host 127.0.0.1 --port 8080 --workers 4 --proxy-headers --forwarded-allow-ips '*'
Restart=on-failure
RestartSec=5
StandardOutput=append:/var/log/amplifi-cleanup/backend.log
StandardError=append:/var/log/amplifi-cleanup/backend-error.log
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ReadWritePaths=/var/log/amplifi-cleanup /var/lib/amplifi-cleanup /opt/amplifi-cleanup

[Install]
WantedBy=amplifi-cleanup.target
```

Equivalent units for `frontend`, `worker` (Celery), `beat` (Celery beat for cron
jobs). The worker unit may declare multiple queues (`-Q default,odata,ml,llm,email`).

A reverse-proxy unit is **not** shipped (admin runs nginx separately and templates
are provided in `infra/nginx/`).

## 18.4 Pre-flight checks

`scripts/preflight.sh` (run by `install.sh` and `upgrade.sh`):

- Python ≥ 3.11, Node ≥ 20, PostgreSQL ≥ 15, Redis reachable.
- `psql` connection to the configured DB (with the configured user).
- Disk free ≥ 5 GB on the storage path.
- `systemd` available (skip on dev hosts).

Non-zero exit aborts the install with a clear message.

## 18.5 Health & ops endpoints (cross-ref §11.11)

- `/healthz` — liveness (process is alive).
- `/readyz` — readiness (DB, Redis, configured LLM provider, SMTP TCP) — **gates
  load-balancer routing**.
- `/metrics` — Prometheus.
- `/admin/jobs`, `/admin/logs/tail`.

Sample Prometheus alert rules are shipped under `infra/prometheus/alerts/` (DB
down, Celery queue backlog, daily cost cap > 80%, error-rate spike).
